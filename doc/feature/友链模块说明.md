# 友链模块说明

本文档说明 **当前项目中友链模块的实际组织方式、文件职责、调用链路、页面表现与维护方式**。

## 1. 模块目标

`Friends` 是站点中的独立内容模块，用于集中展示值得长期互相访问的网站、博客与个人主页链接。

当前实现目标包括：

1. 在顶部导航增加 `Friends` 入口，并放在 `Insights` 右侧。
2. 新增独立页面 `/friends/`。
3. 友链内容通过正式内容链路接入，而不是直接写死在页面中。
4. 友链按分类分组展示。
5. 每条友链以“头像 + 站点名 + 描述”的卡片形式展示。
6. 页面同时适配浅色与深色主题。
7. 页面底部保留统一的 `cd ..` 返回入口，并为友链页单独处理对齐与宽度。

## 2. 当前组织方式

友链模块遵循项目现有的内容组织模式：

```text
src/content/friends/data.json      提供原始友链数据
          ↓
src/content.config.ts              注册 friends collection
          ↓
src/content/schema.ts              校验 friend 数据结构
          ↓
src/utils/data.ts                  排序并分组整理友链数据
          ↓
src/pages/friends.mdx              定义页面入口与页面级 frontmatter
          ↓
src/components/views/FriendsView.astro
                                   负责分组渲染、卡片展示与申请区块
          ↓
BaseLayout + StandardLayout        输出最终页面结构与底部返回入口
```

这意味着它不是“在页面里手写一堆链接”的实现，而是正式进入了项目的 `astro:content` 内容链路。

## 3. 相关文件总览

当前友链相关代码主要涉及以下文件：

| 文件 | 作用 |
| :-- | :-- |
| `src/config.ts` | 在顶部导航增加 `Friends` 入口 |
| `src/content.config.ts` | 注册 `friends` collection |
| `src/content/schema.ts` | 定义 `friendSchema`，约束友链数据结构 |
| `src/utils/data.ts` | 提供友链排序与分组函数 |
| `src/pages/friends.mdx` | `/friends/` 页面入口 |
| `src/components/views/FriendsView.astro` | 渲染友链分组、卡片和申请友链区域 |
| `src/content/friends/data.json` | 存放每条友链的数据 |
| `src/layouts/BaseLayout.astro` | 负责友链页底部 `cd ..` 的宽度与对齐 |

下面按文件说明。

## 4. 各文件作用与内容

### 4.1 `src/config.ts`

作用：

- 在站点顶部导航中加入 `Friends`

当前是在 `UI.internalNavs` 中新增一项：

```ts
{
  path: '/friends',
  title: 'Friends',
  displayMode: 'alwaysText',
  text: 'Friends',
}
```

这使得 `NavBar.astro` 在渲染导航时，会自动把 `Friends` 显示到顶部导航栏中，并按配置顺序出现在 `Insights` 的右侧。

### 4.2 `src/content.config.ts`

作用：

- 把 `src/content/friends/data.json` 接入 Astro 内容系统

当前内容集合形式为：

```ts
const friends = defineCollection({
  loader: file('./src/content/friends/data.json'),
  schema: friendSchema,
})
```

它的意义是：

- 友链数据会被识别成 `friends` collection
- 每条友链都会经过 schema 校验
- 页面和视图层可以通过 `astro:content` 正式读取它们

### 4.3 `src/content/schema.ts`

作用：

- 定义友链数据结构

当前 `friendSchema` 主要字段如下：

- `id`：友链唯一标识
- `name`：站点名称
- `link`：站点链接
- `avatar`：头像图片地址，允许为空字符串
- `desc`：站点描述
- `category`：友链分组
- `siteLabel`：补充说明，默认空字符串
- `order`：同组内排序值，默认 `999`

这个文件解决的问题是：

- 防止字段缺失或写错
- 保证页面读取到的数据结构稳定
- 让后续新增友链时有统一规范

说明：

- 当前友链系统已经彻底移除 `featured`
- 友链是否靠前完全由 `order` 和名称共同决定

### 4.4 `src/utils/data.ts`

作用：

- 提供友链的数据整理逻辑

当前主要函数有两个：

- `getSortedFriends('friends')`
  - 从 `astro:content` 读取友链数据
  - 先按 `order` 排序
  - 最后按名称排序
- `getGroupedFriendsByCategory('friends')`
  - 把所有友链按 `category` 分组
  - 返回页面可直接使用的数据结构

它的职责不是渲染 UI，而是把“原始友链条目”整理成“页面真正需要的数据结构”。

### 4.5 `src/pages/friends.mdx`

作用：

- 定义 `/friends/` 页面的路由入口
- 提供页面级 frontmatter
- 使用项目统一布局挂载友链视图

当前页面 frontmatter 包含：

- `title: 友情链接`
- `subtitle: 记录那些愿意长期互相访问、彼此欣赏的网站与博客`
- `description`
- `bgType: false`
- `ogImage: true`

当前页面结构是：

```text
BaseLayout
  ↓
StandardLayout
  ↓
FriendsView
```

其中：

- 页面使用统一布局，保留站点现有标题、副标题、返回链接和页脚结构
- 页面主体由 `FriendsView.astro` 负责
- 页面本身不负责数据处理，只负责挂载视图组件

### 4.6 `src/components/views/FriendsView.astro`

作用：

- 读取 `getGroupedFriendsByCategory('friends')` 返回的数据
- 渲染友链分组标题
- 渲染头像卡片网格
- 渲染申请友链区域
- 负责浅色与深色主题下的局部样式变量

它负责的页面行为包括：

- 按分类输出多个友链分组
- 每个分组显示标题和友链数量
- 分类之间 **不再使用分割线**
- 每条友链以整卡点击方式跳转
- 卡片中展示头像、站点名、简介和补充说明
- 鼠标悬浮到卡片时，浏览器原生提示展示的是完整 `desc`
- 当头像缺失时，用站点名称首字母作为回退占位
- 页面底部额外提供“申请友链”说明区块

当前样式特点：

- 友链主容器宽度由组件内单独控制
- 桌面端主网格为一行 `4` 列
- 屏幕缩小时依次退化为 `3` 列、`2` 列和 `1` 列
- 页面主容器相对标题做了轻微上移，缩短标题与卡片区的空白
- 页面顶部卡片区域内部说明文案已删除，只保留分组与卡片本体
- 卡片 hover 仅做轻微位移、边框增强和背景变化

它是友链模块的核心视图文件。

### 4.7 `src/content/friends/data.json`

作用：

- 存放每条友链的真实数据

当前每条友链都是一条独立 JSON 记录，适合长期维护和排序。

示例结构：

```json
{
  "id": "anthony-fu",
  "name": "Anthony Fu",
  "link": "https://antfu.me/",
  "avatar": "https://github.com/antfu.png?size=96",
  "desc": "Vue、Vite 与开源生态的长期建设者。",
  "category": "大佬们",
  "siteLabel": "站在巨人的肩膀上",
  "order": 1
}
```

这类数据文件是友链模块最核心的维护对象。

### 4.8 `src/layouts/BaseLayout.astro`

作用：

- 控制所有非首页页面底部 `cd ..` 的输出
- 为 `friends` 页和 `insights` 页分别指定不同的 footer 宽度和对齐方式

当前实现方式：

- 基于 `pathname` 判断 `isFriendsPage`、`isInsightsPage`、`isHomePage`
- 生成统一的 `footerClasses`
- 生成统一的 `footerStyle`

这样做的原因是：

- 项目启用了 `ClientRouter`
- 站内跳转时如果某个页面依赖“移除旧样式”来恢复默认布局，容易出现样式残留
- 当前改成每类页面都显式输出自己的 footer 样式后，`friends -> insights` 的跳转表现更稳定

## 5. 整体模块链路

### 5.1 内容进入系统的链路

```text
src/content/friends/data.json
        ↓
content.config.ts 注册为 friends collection
        ↓
schema.ts 中的 friendSchema 校验字段结构
        ↓
astro:content 提供结构化 entry
```

### 5.2 页面渲染链路

```text
src/pages/friends.mdx
        ↓
FriendsView.astro
        ↓
getGroupedFriendsByCategory('friends')
        ↓
getSortedFriends()
        ↓
输出按分类分组的友链页面
```

### 5.3 导航进入链路

```text
src/config.ts
        ↓
UI.internalNavs 新增 /friends
        ↓
NavBar.astro
        ↓
顶部导航显示 Friends
```

### 5.4 返回入口链路

```text
BaseLayout.astro
        ↓
根据 pathname 生成 footerClasses / footerStyle
        ↓
输出 BackLink 组件
        ↓
页面底部显示 cd ..
```

## 6. 页面最终表现

当前 `Friends` 页面具备以下展示特征：

- 顶部导航可以直接进入 `/friends/`
- 页面采用 `BaseLayout + StandardLayout`
- 页面标题和副标题由 `friends.mdx` 提供
- 友链按分类分组显示
- 每个分组包含标题、数量和卡片列表
- 卡片以“头像 + 站点名 + 描述”作为主要信息结构
- 整张卡片可点击跳转，不依赖额外按钮
- 分类之间不再插入横向分割线
- 鼠标悬浮时浏览器原生提示展示完整描述
- 页面底部提供申请友链区块和单独对齐的 `cd ..`
- 浅色模式下保持轻盈、留白足够的卡片感
- 深色模式下通过边框、背景和文字层级保持可读性

## 7. 维护方式

### 7.1 新增一条友链

在 `src/content/friends/data.json` 中追加一项即可：

```json
{
  "id": "my-friend",
  "name": "My Friend Blog",
  "link": "https://example.com",
  "avatar": "https://example.com/avatar.png",
  "desc": "记录技术、生活与长期主义。",
  "category": "朋友们",
  "siteLabel": "有趣的长期写作",
  "order": 10
}
```

新增后效果：

- 页面会自动读取该数据
- 会自动进入对应分类
- 会按排序规则插入到合适位置

### 7.2 修改友链页面样式

主要修改文件：

- `src/components/views/FriendsView.astro`

这个文件同时包含：

- 分组结构
- 卡片结构
- 页面样式
- 主题变量
- 申请友链区块

因此视觉调整基本都集中在这里完成。

### 7.3 修改字段规范

主要修改文件：

- `src/content/schema.ts`

如果未来想新增字段，例如：

- `rss`
- `location`
- `tags`

应先在 `friendSchema` 中补字段，再在数据文件中使用。

### 7.4 修改排序规则

主要修改文件：

- `src/utils/data.ts`

当前排序优先级为：

1. `order`
2. `name`

### 7.5 修改底部 `cd ..` 的对齐方式

主要修改文件：

- `src/layouts/BaseLayout.astro`

`friends` 页和 `insights` 页的底部返回入口宽度与对齐都在这里统一处理。

### 7.6 修改导航入口

主要修改文件：

- `src/config.ts`

导航文案、顺序、显示方式都由这里统一控制。

## 8. 当前模块特点

相比直接在页面模板里手写一组链接，当前友链模块有几个明显特点：

- 内容进入了项目统一的 `astro:content` 体系
- 数据结构稳定，适合长期维护
- 页面壳、数据处理、视图渲染分层清晰
- schema 可以约束内容格式，降低维护成本
- 页面默认适配站点浅色与深色主题
- 页面底部返回入口已经做了页面级布局适配
- 后续新增内容时，不需要修改页面逻辑

## 9. 一句话总结

当前友链模块已经成为项目中的一条正式内容链路：

- `src/content/friends/data.json` 负责存内容
- `content.config.ts` 和 `schema.ts` 负责接入与校验
- `utils/data.ts` 负责排序与分组
- `src/pages/friends.mdx` 负责页面入口
- `FriendsView.astro` 负责最终友链卡片与申请区块展示
- `BaseLayout.astro` 负责底部 `cd ..` 的页面级布局控制
