# 减少非官方依赖开发更新说明（仅实现阶段一二）

本文档面向当前项目的依赖治理，目标不是盲目“减少依赖总数”，而是优先减少 **非主栈官方维护、可被内建能力或项目自研替代** 的依赖，从而降低升级风险、维护成本与生态耦合。

## 1. 判断标准

为了避免“官方 / 非官方”定义过于模糊，本文档采用下面这套口径：

- **主栈官方**：由当前主链路技术栈的核心组织维护，例如 Astro、ESLint、TypeScript、UnoCSS 官方维护的包。
- **生态官方**：不是 Astro 官方，但属于其所在生态的上游官方项目，例如 unified/remark/rehype、Pagefind、Sharp、Satori 等项目的官方包。
- **社区第三方**：由独立作者或社区维护，不属于当前主栈核心组织，也不属于当前直接依赖链上的上游官方实现。

本次“减少非官方依赖”的治理重点，默认优先处理第三类，其次再评估第二类里是否存在可以用内建能力替代的包。

## 3. 代码库中的实际使用情况

### 3.1 明确存在使用、暂时不能直接删除

1. `remark-directive-sugar`
   - 在 `src/content/home/index.md` 中存在多处 `:link[...]` 语法。
   - 这说明它不是“装了没用”，而是已经进入内容表达层。

2. `rehype-callouts`
   - 在多篇文章中存在 `> [!NOTE]`、`> [!IMPORTANT]` 语法。
   - 如果直接移除，会导致历史文章提示块渲染退化。

3. `remark-math` + `rehype-katex`
   - `src/content/blogs/markdown-guide.md` 中存在 `$...$` 与 `$$...$$`。
   - 数学公式能力当前有真实使用。

4. `astro-expressive-code` 及其两个插件
   - 已在 `astro.config.ts` 和 `ec.config.mjs` 中完成完整接入。
   - 当前代码块展示依赖这一整条链路。

5. `nprogress`
   - 在 `src/layouts/BaseLayout.astro` 中直接引入并用于页面切换进度条。

6. OG 生成链
   - `plugins/remark-generate-og-image.ts` 直接使用 `satori`、`satori-html`、`sharp`、`html-entities`、`chalk`。
   - 这是一条真实运行中的构建能力，而不是备用代码。

### 3.2 已经出现“可直接清理”信号的依赖

1. `p5`
   - 当前代码库未检索到 `p5` 的导入、实例化或直接调用。
   - 若没有运行时通过 CDN 或隐藏脚本注入的情况，可以优先删除。

2. `@types/p5`
   - 与 `p5` 绑定。
   - 若 `p5` 删除，该类型包也应同步删除。

3. `remark-imgattr`
   - 当前内容目录未检索到 `![...](...){...}` 这类图片属性扩展语法。
   - 有较大概率属于遗留依赖。

### 3.3 已经具备“可迁移后删除”条件的依赖

1. `astro-robots-txt`
   - 当前需求本质只是输出 `robots.txt`。
   - 对纯静态博客来说，直接维护 `public/robots.txt` 的复杂度更低。

2. `del-cli`
   - 当前只用于 `postbuild` 中删除 `dist/pagefind/*ui*`。
   - 可以改成一个本地 Node 脚本，减少额外 CLI 依赖。

3. `reading-time`
   - 当前已经有自定义 `remark-reading-time.ts`。
   - 既然已有本地插件，阅读时长计算完全可以内聚进项目自身。

4. `nprogress`
   - 只有一个页面切换进度条用途。
   - 可改成更轻量的原生顶部 loading bar，或直接取消该效果。

5. `rehype-wrap-all`
   - 当前只用于把 `table` 包成 `div`。
   - 这是很小的 DOM 变换，完全可以本地实现。

6. `satori-html`
   - 当前只是 OG 模板桥接层。
   - 若后续继续保留 OG 自动图生成功能，可以改为直接构造 Satori 节点树，减少一层第三方。

## 4. 建议的减依赖优先级

### A 档：优先立即处理

这些依赖要么疑似未使用，要么替换成本极低：

| 依赖 | 建议动作 | 预期收益 |
| :-- | :-- | :-- |
| `p5` | 检查后直接删除 | 立即减少无效依赖 |
| `@types/p5` | 随 `p5` 一起删除 | 清理无效类型依赖 |
| `remark-imgattr` | 验证后删除 | 简化 Markdown 插件链 |
| `astro-robots-txt` | 改为 `public/robots.txt` | 降低 Astro 第三方集成数量 |
| `del-cli` | 改为本地 Node 脚本 | 降低 CLI 型第三方依赖 |

### B 档：建议本迭代处理

这些依赖虽然在用，但替换收益明显，且改造面可控：

| 依赖 | 建议动作 | 说明 |
| :-- | :-- | :-- |
| `reading-time` | 内聚到本地插件 | 当前已有自定义 remark 插件承载点 |
| `nprogress` | 自研轻量 loading bar 或移除 | 只服务一个小交互点 |
| `@types/nprogress` | 随 `nprogress` 去留同步处理 | 联动清理 |
| `rehype-wrap-all` | 改成本地 rehype 插件或样式策略 | 逻辑简单，替代容易 |
| `chalk` | 改为普通日志 | 构建日志不需要彩色输出也可接受 |

### C 档：建议在内容迁移后处理

这些依赖绑定了既有内容语法，不能粗暴移除：

| 依赖 | 建议动作 | 说明 |
| :-- | :-- | :-- |
| `remark-directive-sugar` | 先迁移 `:link[]` 等语法，再移除 | 首页内容已使用 |
| `rehype-callouts` | 先统一提示块写法，再移除 | 历史文章已有 `[!NOTE]` 风格内容 |
| `astro-expressive-code` | 评估是否降级到 Astro 默认代码高亮 | 影响所有代码块展示体验 |
| `@expressive-code/plugin-line-numbers` | 随上项统一决策 | 属于附属依赖 |
| `@expressive-code/plugin-collapsible-sections` | 随上项统一决策 | 属于附属依赖 |
| `satori-html` | 若保留 OG 自动图，再做重构替换 | 不属于首批治理对象 |

## 5. 推荐实施路线

### 阶段一：先删无争议依赖

建议先完成下面这一组：

1. 删除 `p5` 和 `@types/p5`
2. 验证并删除 `remark-imgattr`
3. 用 `public/robots.txt` 替换 `astro-robots-txt`
4. 用本地 Node 脚本替换 `del-cli`

这一阶段的特点是：

- 风险低
- 回报快
- 对内容层和 UI 层影响最小

### 阶段二：收缩“小而散”的第三方工具

建议继续处理：

1. 将 `reading-time` 逻辑内聚到本地
2. 去掉 `nprogress` 与 `@types/nprogress`
3. 本地化 `rehype-wrap-all`
4. 将 `chalk` 从 OG 构建日志中移除

这一阶段的目标，是把“一个小需求对应一个第三方包”的情况尽量减少。

### 阶段三：治理内容语法耦合

最后再处理：

1. `remark-directive-sugar`
2. `rehype-callouts`
3. `astro-expressive-code` 及其附属插件
4. `satori-html`

这一阶段不能只做代码改动，还需要同步考虑：

- 历史文章兼容
- 首页内容改写成本
- 代码块展示风格是否接受退化
- OG 图生成链是否继续保留

## 6. 建议保留的依赖

为了避免“为了减依赖而减依赖”，下面这些依赖虽然不是 Astro 官方，但当前阶段建议保留：

- `pagefind`
  - 当前搜索功能明确依赖它，替代成本高。
- `remark-math` 与 `rehype-katex`
  - 当前内容存在真实公式示例。
- `@iconify/json`
  - 当前 UnoCSS 图标能力依赖它。
- `sharp`
  - 如果保留 OG 自动图或后续图片处理，仍是高价值基础依赖。
- `rehype-autolink-headings` 与 `rehype-external-links`
  - 虽可自研，但现在收益不高，不应优先处理。

## 7. 最终建议

如果本轮工作的目标是 **“减少非官方依赖，但不显著增加维护风险”**，建议把实际执行目标收敛为：

1. **本轮立即删除**
   - `p5`
   - `@types/p5`
   - `remark-imgattr`（确认无回归后）

2. **本轮替换**
   - `astro-robots-txt` -> `public/robots.txt`
   - `del-cli` -> 本地 Node 构建后处理脚本
   - `reading-time` -> 项目内实现
   - `nprogress` / `@types/nprogress` -> 更轻实现或直接移除
   - `rehype-wrap-all` -> 本地最小插件

3. **下轮再处理**
   - `remark-directive-sugar`
   - `rehype-callouts`
   - `astro-expressive-code`
   - `@expressive-code/plugin-line-numbers`
   - `@expressive-code/plugin-collapsible-sections`
   - `satori-html`

一句话总结：

> 当前项目里真正应该优先减少的，不是 Astro / ESLint / TypeScript / UnoCSS 这类主栈官方依赖，而是那些 **只为单点功能服务、且可以被静态文件、本地脚本、自定义插件或原生实现替代** 的社区第三方包；其中 `remark-directive-sugar` 正属于“明确不是官方、且后续值得迁移掉”的代表性依赖。
